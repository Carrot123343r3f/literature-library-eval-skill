# 输入协议（状态机）

## 固定首轮开场（“评估我的库”）

首轮最多问三个问题；不展示后续批次清单，也不承诺尚未实现的直连能力。可直接使用下列话术（已知信息则跳过）：

> 我可以先判断你的文献库是否足以支撑目标综述，并把检索式、迭代过程和下一步建议写进报告。请告诉我三件事：
> 1. 你的研究问题/边界，以及打算写哪类综述（叙事、系统、范围、快速或伞式）？
> 2. 文献库在哪里？当前脚本完整支持 JSON；Zotero、BibTeX、RIS、CSV 可以先由 AI 协助规范化为 JSON，但还不是直接导入。
> 3. 你已有检索式、必纳入文献或权威综述/标准吗？有的话我保留它们作为起点；没有的话我会主动建立“待核验”的锚点候选与首轮检索诊断。

第一轮后再按缺口逐步索取材料；一次仍不超过三个问题。

任何 AI 在处理文献库评估时，必须严格按以下状态机执行。核心原则：**先判断用户属于哪一类，再索取最小必要信息**——不要一上来就列全量四批次。

## 状态机流程

```
用户触发
  │
  ├─→ [S0] 识别用户输入形态
  │     │
  │     ├─ 只给题目/研究问题 ──→ [S1] 提取问题要素 → 简短确认 → 继续 S2
  │     ├─ 只给工程领域 ──→ 询问拟写的综述问题与范围 → 返回 S0
  │     ├─ 给题目和文献库 ──→ [S4] 库格式检查 → 继续 S2
  │     ├─ 给完整检索式/日志 ──→ [S4] 复用并审计现有过程
  │     ├─ 给 Zotero/CSV/BibTeX/JSON ──→ [S4] 识别格式并规范化
  │     ├─ 只说"评估我的库" ──→ 询问库位置、综述类型、研究问题（批次一）→ 返回 S0
  │     ├─ 题目不清楚 ──→ 生成 2-3 个可选的工程综述问题表述，要求确认
  │     └─ 超出范围 ──→ 不启动 A-F 工程充分性审计 → [S_停止]
  │
  ├─→ [S2] 范围路由
  │     ├─ 明确工程领域 ──→ scope_status = "in_scope" → 继续 S3
  │     ├─ 工程交叉 ──→ scope_status = "cross_domain" → 说明适用/限制部分 → 继续 S3
  │     ├─ 纯数学/纯物理/纯化学/临床医学/基础生命科学 ──→ scope_status = "out_of_scope" → [S_停止]
  │     └─ 无法判断 ──→ scope_status = "scope_uncertain" → 只问一个关键澄清问题 → 返回 S2
  │
  └─→ [S3] 最小必要确认（见下方三层确认）
        │
        ├─ 必须确认项（逐项，不可跳过）
        │   1. 研究问题/题目
        │   2. 综述类型
        │   3. 文献库位置（或是否允许新建检索）
        │   4. 时间与语言边界
        │   5. 是否允许自动检索、调词和生成候选
        │   6. 输出语言与报告格式
        │
        ├─ 可选补充（主动提示但不追问）
        │   · 已知必纳入文献
        │   · 已知核心关键词
        │   · 分类框架
        │   · 指定数据库
        │   · 已知缺口
        │   · 用户阈值偏好
        │
        └─ 输出 → run-config.json → 后续流程只读配置，不重新依赖聊天上下文
```

## S_停止：超出范围时的行为

当 `scope_status = "out_of_scope"` 时：

1. 说明工程范围定义和研究问题为何不符合——语气是解释，不是拒绝。
2. 提供降级服务选项：
   - **题录健康检查**：元数据完整性（F2）、去重（F4）、基础时效描述（D2 仅统计不判标准）、来源分布（C2 仅描述不判均衡）
   - **检索准备度设计**（若无库但有题目）：生成检索策略、数据库列表、纳入标准草案
   - 建议用户考虑替代评估框架（如临床问题的 Cochrane/AMSTAR 路径，纯理论学科不作推荐）
3. 不生成 A–F 工程充分性结论。
4. 记录在 `run-config.json.scope_status` + `scope_rationale`。
5. 范围外回复的语气应有助于用户理解"什么可以、什么不可以"，并在他们愿意时切换到降级服务。

当 `scope_status = "cross_domain"` 时：

- 标记工程适用部分和非适用部分。
- 对适用部分运行完整 A-F。
- 对非适用部分降级为仅题录健康或跳过，并在报告中分段说明。

## 三层标准确认

### 第一层：默认采用

一次性告知默认标准：

> 本次按"[综述类型] + [工程领域]"默认标准执行，共 22 项（伞式 25 项）。各项含义、计算和阈值见《标准说明书》。你有以下选择：

### 第二层：展示影响结论的核心标准

只展示少数几项真正影响结论的门槛：

- A1/A2 的召回阈值（seminal/gold 的找回比例）
- B1/B2 的趋稳规则（GGR/DRR 的收敛判据）
- D2 的近年窗口和比例（时效诊断）
- F2/F3/F5 的可用性阈值（元数据/全文/谱系）
- C4 观点偏斜度：需声明中心主张，并检查支持、质疑与条件性证据是否失衡
- 伞式综述时：A4/C5/F7

### 第三层：给用户选择（四选一）

| 选项 | 行为 |
|---|---|
| 使用推荐默认标准 | 全量采用 profile + 综述类型的默认值 |
| 按我的领域调整 | 用户指定 profile 或具体阈值→合并进 `run-config.json.standards.user_overrides` |
| 我给出自己的标准 | 用户提供自定义 `standards` JSON→直接使用（校验格式） |
| 仅报告数据，不做阈值判定 | `standards.confirmed_by_user = false`——所有判定列为 `screening`，由人工解读 |

## 用户友好性要求

以下是 AI 在执行输入协议时必须遵循的交互原则：

1. **用户不需要理解任何技术术语**。不要要求用户提供 Gold set、去重日志、来源快照、run-config——这些都是后台产物。你应说"有没有你确定必须引用的论文？"而不是"请提供 A1 基准集 JSON"。
2. **一次最多问三个问题**。超过三个时，拆成多轮。每轮告知"这组确认后你就可以去做别的事了，我后台继续跑"。
3. **对不确定的综述类型，给出推荐并解释原因**，而不是罗列五种类型让用户选。
4. **`not_assessable` 必须附带"最低成本补什么"**。不能只说"不可评估"，要加一句"如果你想补，最简单的方式是给我 X"。
5. **范围外不拒绝，提供降级护航**："这不属于工程问题的充分性审计范围，但我可以帮你做这些：[题录健康/检索设计/……]。你想做哪个？"
6. **`run-config.json` 是后台产物**，不在确认阶段向用户展示——生成后告知"配置已保存"即可。

## 特殊情形分支（必须实现）


### 1. 没有文献库

- `library.provided = false`
- 允许执行"检索准备度设计"：生成检索策略、数据库列表、纳入标准草案→输出为检索计划，不输出库评估。
- 报告中注明：**"本次未提供文献库，仅输出检索准备度设计，未做文献库评估。"**

### 2. 没有 Gold/benchmark

- 默认运行首轮编排器：以多源候选锚点形成 A1 自动初筛、机械留出集形成 A2 自动初筛、多源快照形成 A3 下界；它们必须标明证据等级，不能作为实测或饱和结论。
- 绝不从**被测 q\*** 的检索命中反向构造基准集（会导致循环论证）。候选锚点必须保留独立的来源快照与构建记录。
- AI 仍应主动询问用户是否有 3–5 篇明确必须引用的论文；用户锚点经核验后优先于自动候选。

### 3. 没有筛选决定

- B 只能报告候选发现（`discovery_candidates` 和 `discovery_ggr`）。
- 不能报告 GGR 饱和状态或趋稳结论。
- 报告中注明："**B 维处于候选发现阶段——尚未完成筛选，饱和度不可评估。**"

### 4. 只有单一数据库/来源

- 若用户限制单一来源，A3 设定为 `not_assessable`——单源不能推断多源覆盖；B2/B3 仍要显示，但只说明该限制导致无法判断饱和。
- C2 可能偏向该单一来源→在报告中标注覆盖边界。

### 5. 检索源访问失败

- 保存失败日志到 `run-config.json.evidence_inputs.failed_sources`。
- 继续可用来源并从已有结果中计算。
- 报告中注明哪些指标因来源失败而降为 `partial_snapshot` 或 `not_assessable`。

### 6. 用户限制只用某数据库

- 接受限制。
- 报告中标注："**按用户要求仅使用 [数据库名称]，覆盖边界受限于该数据库的收录范围。**"
- A3/C2 的判定应反映该限制。

### 7. 用户不愿自动检索

- 仅审计现有库。
- A2 的实测部分标 `not_assessable`（无在线检索命中）。
- B 标 `not_assessable`（无多轮数据）。
- 报告中注明："**按用户选择未执行自动检索，A2/B 基于现有数据（如已提供）计算。**"

### 8. 文献库很小（如 <10 篇）

- 不以"样本量小"直接判定 `fail`。
- 区分窄问题（少数文献是正常的）与检索不足（应该有但没有）。
- 若为窄问题：报告中注明"库规模小但可能反映窄研究问题边界"。
- 若为检索不足：A1/C1 可能发现样本小但缺失覆盖面。

### 9. 领域不在范围

- 见上方 S_停止。

### 10. 题目不明确

- 生成 2-3 个可能的工程综述问题表述（基于关键词/领域推断）。
- 要求用户确认或修改，不进入正式流程。
