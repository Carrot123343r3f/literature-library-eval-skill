# 审计指标字典：名称、含义与计算方法

本文件是 `audit.md` 中 A1–F8 的权威解释。每行均应报告采用标准、原始值、证据状态和行动；`not_assessable` 表示没有足以作出该项结论的输入或可复跑证据。

## A 覆盖验证

| 子项与报告名称 | 它回答什么 | 计算或核验方法 |
| --- | --- | --- |
| A1 已知必纳入文献找回率 | 已确认应纳入的文献是否已经在库中 | `库∩基准集 / 基准集`；仅以 DOI、PMID、PMCID、arXiv、OpenAlex 等稳定 ID 匹配。阈值由 `a1_min_recall` 配置。 |
| A2 检索式找回已知相关文献的能力 | 已执行检索式是否能找回独立 gold set | `检索命中∩gold set / gold set`；零命中为实测 0，阈值由 `a2_min_recall` 配置。 |
| A3 多来源检索到的最少唯一候选数 | 已检索来源至少提供多少不同候选 | 对完整来源快照按稳定 ID 去重后取并集；这是下界，不是 Recall。 |
| A4 关键标准和核心文献是否找全 | 项目指定的标准、指南或奠基文献是否命中 | 对用户给定锚点清单逐条稳定 ID/标准号核验。当前脚本未接收该清单，故为不可评估。 |

## B 范围结构

| 子项与报告名称 | 它回答什么 | 计算或核验方法 |
| --- | --- | --- |
| B1 工程问题的分类框架是否完整 | 研究问题是否被拆成可检查的工程切面 | 检查 taxonomy 是否定义工况、方法、指标、应用等预期层及规则。 |
| B2 每个关键工程场景是否有文献 | 是否存在明确的场景/方法/指标缺口 | 每个预期层的高置信记录数；默认至少 1 条，系统综述可设 2 条。 |
| B3 文献分类结果是否可信 | B1/B2 是否建立在可靠分类上 | 每层分类置信度 `≥0.80`，未分类率 `≤10%`；应保留规则或复核样本。 |
| B4 是否过度依赖单一来源 | 检索是否被单一数据库/来源主导 | 报告来源计数、最大来源占比和 Shannon entropy；最大占比 `>80%` 警示。 |
| B5 关键场景是否逐项验证找回率 | 小而关键的层是否发生检索盲点 | 每个关键层分别构建基准/Gold set 后计算 A1/A2。 |

## C 工程证据适配

| 子项与报告名称 | 它回答什么 | 计算或核验方法 |
| --- | --- | --- |
| C1 所需工程证据类型是否齐全 | 是否缺仿真、台架、现场、标准测试等必要证据 | 必需证据类型集合与记录 `evidence_type` 集合求差；每类至少 1 条高置信记录。 |
| C2 文献是否说明适用工况 | 结论能否放回实际部署边界 | 有 `operating_conditions` 或 `test_conditions` 的记录比例；默认 `≥80%`。 |
| C3 文献是否给出可比的指标和基线 | 性能结论能否横向对照 | 有 `baseline` 或 `performance_metrics` 的记录比例；默认 `≥80%`。 |
| C4 标准、数据和软件版本是否明确 | 比较、合规和复现是否可靠 | 核验标准、数据集和模型/软件版本；当前由结构化 verdict 输入。 |
| C5 验证是否足以支撑工程结论 | 仿真、实验和现场证据是否被恰当外推 | 分别报告验证类型与数量，按项目验证层级核验；当前由结构化 verdict 输入。 |
| C6 可信度风险是否已处理 | 预印本、撤稿、更正、代码/数据线索是否已处置 | 自动统计标记；关键结论再逐条人工/专门来源核验。 |

## D 时效与技术演化

| 子项与报告名称 | 它回答什么 | 计算或核验方法 |
| --- | --- | --- |
| D1 各检索来源是否仍是最新 | 每个来源最近成功检索是否在允许窗口内 | 用今天减去每个 `last_successful_search` 日期；默认 `≤90` 天，可按领域改为 30/180。 |
| D2 计划检索的来源是否全部完成 | 是否有计划来源未跑或失败 | `planned_sources` 与成功检索日期集合逐项比较。 |
| D3 新兴技术和新术语是否被覆盖 | 前沿窗口是否有专门验证 | 对最近窗口做 A2 或完整多源检索；当前由结构化 verdict 输入。 |
| D4 是否使用了最新且正确的文献版本 | 是否误用预印本、会议版或过时标准 | 核验预印本—正式版、在线优先—刊出版和标准版本；当前由结构化 verdict 输入。 |
| D5 文献年代结构是否已被检查 | 历史基础与近期发展是否可见 | 报告年份分布、中位发表年龄和近窗比例；不设跨领域统一配额。 |

## E 学术影响与知识关联

| 子项与报告名称 | 它回答什么 | 计算或核验方法 |
| --- | --- | --- |
| E1 引用数据是否足够用于背景判断 | 影响力背景是否有来源和日期 | 有 `cited_by_count` 的记录比例；默认 `≥80%`。 |
| E2 影响力是否由少数文献主导 | 是否仅被少数高被引记录带偏 | 报告中位数、年龄/领域归一化分位及长尾；当前脚本至少报告中位数。 |
| E3 是否连到关键研究和技术路线 | 是否与关键方法、标准、应用脱节 | 核验前向/后向引文路径，必要时共同引文或参考文献重叠；当前由结构化 verdict 输入。 |
| E4 是否过度依赖单一发表渠道 | 渠道偏差是否明显 | 报告来源、venue 和文献类型分布；单一渠道 `>80%` 警示。 |
| E5 领域标准和权威工作是否齐全 | 工程共识节点是否缺失 | 按项目权威锚点清单核验命中率；当前由结构化 verdict 输入。 |

## F 过程合规、库健康与可追溯

| 子项与报告名称 | 它回答什么 | 计算或核验方法 |
| --- | --- | --- |
| F1 检索过程能否被他人复跑 | 查询、日期、字段、来源、过滤器是否被保存 | 检查完整 run log；目标为计划查询完整记录率 `100%`。 |
| F2 计划的检索路径是否完成 | 数据库、前向、后向等路径是否跑完 | `完成路径数 / 计划路径数`，目标 `100%`。 |
| F3 检索是否有趋于饱和的证据 | 继续检索的新增收益是否很低 | 连续两轮高置信新增率 `<2%`、新来源收益 `<5%`、路径完成且独立验证通过。 |
| F4 每篇纳入文献能否追溯来源和决定 | 为什么纳入/排除是否可解释 | 有来源谱系的记录比例；默认 `≥95%`，并应保存纳入/排除理由。 |
| F5 题录信息是否完整可用 | 是否能可靠筛选、去重和引用 | 标题、作者、年、载体、稳定 ID 填充率默认 `≥95%`；摘要 `≥80%`。 |
| F6 重复和不同版本是否已妥善处理 | 是否重复计数或静默误删 | 未处置 DOI 精确重复必须为 0；题名相似、预印本—正式版均要有决定。 |
| F7 全文、代码和数据是否有获取线索 | 是否具备获取和复现准备度 | 有附件或开放链接的比例默认 `≥80%`；另报告代码/数据/协议线索。 |
| F8 撤稿、更正和异常是否已核查 | 是否仍在使用已更正或撤回的证据 | 对关键记录保存更正检查日期、来源和处置；目标 `100%`。 |
