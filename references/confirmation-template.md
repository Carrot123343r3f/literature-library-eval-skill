# 后续确认模板（不是固定首轮开场）

> 首轮开场必须使用 `intake-protocol.md` 的“固定首轮开场”，一次最多三个问题。本文件仅用于首轮后按需补充信息；其中的历史格式/连接选项不代表当前脚本具备直接导入能力。

**给 AI 的指令**：首轮之后才使用本模板。根据用户已提供的信息跳过已知项，只问当前流程确实需要的缺口；一次最多三个问题。不要把本文件的全部批次一次性展示给用户。

---

## 批次一：必须拍板（逐个确认）

这些是启动评估的必要输入，缺少任一个无法开始。

### 1.1 研究问题与研究边界

**要收集的**：一句话的核心研究问题；纳入/排除的时间范围（如 2020–2026）；语言限制（默认：英文）；文献类型限制（默认：不限）

**示例提问**：
> 你的研究问题是？有时间和语言限制吗？时间我暂按 XXXX–YYYY，语言按英文。

**智能默认**：时间取"过去 5 年"或"领域爆发期"（如 3DGS 从 2023）；语言默认英文。

### 1.2 综述类型

**要呈现的**：5 种类型 + 简要说明 + 推荐

| 类型 | 一句话 | 何时选 |
|---|---|---|
| 叙事综述 (推荐) | 给出有代表性的工程脉络与论证 | 写"领域进展"或"技术综述"时 |
| 系统综述 | 对工程问题作可复现、低偏倚的综合 | PRISMA 流程、需要检索附录时 |
| 范围综述 | 绘制工程问题的范围和证据图谱 | 摸清"这个方向有哪些子问题"时 |
| 快速综述 | 在限定时间下提供及时的工程证据 | 时间紧迫、需要快速决策时 |
| 伞式综述 | 综合已有工程综述 | 领域已有大量综述、需要整合时 |

**智能默认**：用户说"写综述"/"评估文献库"→叙事综述；说"系统检索"/"PRISMA"→系统综述。

### 1.3 文献库位置与格式

**要收集的**：库在哪、什么格式、大约多少条

**当前可执行的场景**：
- JSON 题录文件（完整支持；供 `run_audit.py` 直接读取）
- Zotero/BibTeX/CSV/RIS/PDF 目录（可由 AI 协助整理、规范化为 JSON；不是脚本的直接导入器）
- 只有题目（可先构建检索与候选锚点计划，A–F 会如实保留缺失项）

**智能默认**：若 MCP 或环境变量已配 Zotero→自动推荐；否则问路径。

## 批次二：运行策略（给推荐默认，可一键跳过）

### 2.1 自主策略

- **全自动**（推荐）：自动冻结高/低置信度集合，仅在缺少必要输入或范围实质变化时暂停
- **保守自动**：仅高置信度进入核心指标，低置信度候选留待人工复审
- **人工终审**：每个阶段完成后暂停等待确认

### 2.2 数据源

- **开放优先 + 在线补全**（推荐）：无用户锚点或检索式时，首轮通过 `run_initial_assessment.py` 自动建立多源候选锚点、生成 q0 + 原子变体，并给出 A1–A3/B1–B3 的初评。首轮和后续轮次使用相同判定：A1/A2/B1 直接按阈值输出，B2 提示独立路径尚缺，B3 在路径或独立验证未完成时直接判不通过；`automated-screening` 只说明证据来自自动初筛。它不完成引文追踪、标准/指南路径或正式筛选，首轮不能声称检索已趋稳。
- **仅离线/本地**：只用题录元数据 + 本地 pdf，不调在线 API（E 维标不可评估）
- **仅指定来源**：严格按用户列出的来源

### 2.3 纳入边界与范围推断

- **自动推断 + 用户确认**（推荐）：AI 根据研究问题推断关键词/时间/领域，用户确认
- **严格按用户给出**：完全遵循用户给出的纳入/排除标准
- **先生成建议再确认**：AI 先生成范围建议，用户审阅后再执行

### 2.4 输出格式

- **Markdown + JSON**（推荐）：`audit.md` 人读 + `audit.json` 机读
- **加 HTML**：额外 `audit.html`
- **加 PDF**：额外 `audit.pdf`（需 pandoc/weasyprint）

**AI 应对批次二**：若用户说"直接开始"或"评估我的库"——全用推荐默认值，不回问批次二。

## 批次三：补充材料（可选但很有价值）

**AI 应主动提示一次**，用户可跳过。这些材料能显著提升基准集质量和覆盖判定精度：

### 3.1 已知必纳入文献

> 有没有你确定"综述必须引用"的论文？给标题/DOI/arXiv ID 即可。

**价值**：直接作为 A1 基准集的种子，比在线推断的 seminal 集更贴合你的研究视角。

### 3.2 已有关键词或检索式

> 有无之前用过的检索词、同义词列表或检索策略？

**价值**：跳过关键词构建阶段，直接进入 gold-set 迭代。

### 3.3 已知缺口或薄弱方向

> 有没有你已经感觉到"文献不够"或"没搜全"的方向？

**价值**：在 C1 主题平衡诊断中标记为"已知缺口"，与自动发现的空主题区分。

### 3.4 分类体系 / Taxonomy

> 有没有现有的分类框架或预期子方向列表？

**价值**：直接作为 taxonomy 输入，比 AI 从标题推断的更精确。

### 3.5 已知的权威综述或标准

> 有没有你认为"这个方向的权威综述/标准/基准数据集"？

**价值**：构建 gold set 和 Tier-1 venue 映射的锚点。

**AI 行为**：列完批次三后说"以上补充材料可直接跳过，不影响基础评估；有的话会提高精度。"

## 批次四：合法性与限制声明（必须告知，不征求确认）

- 订阅源仅在已有合法访问方式时启用
- 自动结果不保证绝对穷尽
- 报告会把实测、估计、自动判定与不可评估分开
- 不要求用户提交 API key；只使用 MCP、环境变量或已配连接
- 不在报告或日志中写入密钥、令牌或受限全文

## 确认后的行为

- 将确认结果汇总为 `context` 输入（字段映射见下表），立即进入自动状态机（见 `autonomous-run.md`），不再为常规步骤提问
- 仅以下情况暂停：缺少必要输入、指定来源不可访问、新发现会实质改变范围

### context 字段映射（确认结果 → `context.json`）

报告的"评估方法与过程"段从这些字段读取，agent 必须把确认结果与执行决策写入对应字段：

| context 字段 | 来源批次 | 含义 | 报告中的体现 |
|---|---|---|---|
| `review_type` | 1.2 | 综述类型 | 基本信息 + 综合判断 |
| `profile` | 1.1 推断 | 工程领域（影响 D2 阈值） | 基本信息 |
| `scope` / `year_start` / `year_end` | 1.1 | 研究范围 | 基本信息 |
| `library_name` / `library_path` | 1.3 | 库位置 | 基本信息·评估对象 |
| `keywords` | 2.3 推断 / 3.2 | 检索关键词列表 | 评估方法·检索关键词 |
| `queries` | 执行时生成 | `[{source, query, date, hits}]` 检索式记录 | 评估方法·检索式 |
| `benchmark_method` / `benchmark_source` / `benchmark_note` | A1 构建时 | 基准集怎么定的 | 评估方法·A1 基准集构建 |
| `gold_set_method` | A2 构建时 | Gold set 怎么定的 | 评估方法·A2 Gold set 构建 |
| `search_sources` / `planned_sources` | 2.2 | 数据来源列表 | 评估方法·数据来源 |
| `failed_sources` | 执行时记录 | 检索失败的来源 | 评估方法·数据来源（括注） |
| `taxonomy` | 3.4 / 推断 | `[{name, expected, high_confidence_records, target_share}]` | 评估方法·分类体系 + C1 计算 |
| `standards` | 2.2/2.3 | 各项阈值（见 `engineering-standards.md`），默认按综述类型分层（叙事 A1≥0.75 / 系统 A1≥0.90 / 快速 A1≥0.60 / 伞式 A1≥0.90） | 总表·标准列 |
| `tier1_venues` | 3.5 | Tier-1 venue 关键字列表 | E2 计算 |
| `last_successful_search` | 执行时 | `{source: date}` 各来源检索日期 | D1 计算 |
| `search_rounds` / `planned_pathways` / `source_marginal_yields` / `independent_validation_passed` | 执行时 | B 饱和度过程数据 | B1/B2/B3 计算 |

未提供的字段 → 对应评估项标 `not_assessable`，报告如实说明缺失输入。
