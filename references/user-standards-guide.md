# 标准说明书（面向用户）

这份说明书不是给 AI 的执行清单——它给你解释每一个标准从何而来、为什么采用这个数值、什么时候该调整。所有默认标准都不是绝对真理；你可以在首次交互中覆盖任意一项。

## 阅读约定

每项按统一格式展开：

- **易懂名称与编号**
- **它回答的问题**：为什么该指标影响"能否支撑综述"
- **计算/核验**：用什么输入、怎么算
- **默认标准**：数值及适用的综述类型 / 工程领域
- **标准依据**：来自工程文献检索经验、方法学约束还是领域惯例
- **何时可调整**：哪些场景应当用不同值，谁来确认，理由记录在哪
- **不达标意味着什么**：风险信号，不是盖棺"库不合格"
- **不可评估时怎么办**：缺哪些输入，是否还能继续其他维度
- **证据类型**：实测 / 估计 / 自动初筛 / 人工核验 / 不可评估

## A 覆盖——已知文献是否找得回来

### A1 基准集召回率

| 项目 | 内容 |
|---|---|
| 易懂名称 | 已知必纳入文献的找回率 |
| 它回答的问题 | 你明确知道"综述必须引用"的那些论文，库里有吗？ |
| 计算/核验 | `库 ∩ 基准集 / 基准集`，只按 DOI/arXiv/PMID/OpenAlex ID 等稳定标识符匹配。标题相似只进人工核验队列，不计入分子。 |
| 默认标准 | 叙事/范围：≥ 0.75；系统：≥ 0.90；快速：≥ 0.60；伞式：≥ 0.90（基准集 ≤ 15 篇时须 = 1.0） |
| 标准依据 | 系统综述方法论对"完整检索"的实证约定（检索漏掉 10% 以上的已知核心文献通常意味着检索策略需要修订）；叙事综述允许 ≤ 25% 的遗漏——它是洞察驱动的，不要求穷尽。 |
| 何时可调整 | 你的研究问题非常窄（如单一算法变体比较）时，可能只要 5 篇锚点就够了→应调低阈值。主导教师/导师要求更严格的证明→调高。调整需在 context 中记录理由。 |
| 不达标意味着什么 | 你的库可能缺少该领域的基石论文。这不等于"综述不能写"，但综述中应该承认这些缺失，或说明它们不在你的视角范围内。 |
| 不可评估时怎么办 | 提供至少 3 篇"我明确知道该引用"的论文——AI 可以此为基础构建基准集。若完全拿不出锚点，A1 标为 `not_assessable`，这不影响其他维度。 |
| 证据类型 | 实测（有稳定 ID 逐条匹配记录）；不可评估（无基准集） |

### A2 检索式灵敏度

| 项目 | 内容 |
|---|---|
| 易懂名称 | 检索式找回已知关键文献的能力 |
| 它回答的问题 | 你用的检索词能在数据库中命中有多少已知相关工作？首轮可由 AI 构建机械留出的候选集给出自动初筛；没有用户或外部来源冻结的验证集时，它不能成为实测结论。 |
| 计算/核验 | `检索命中 ∩ Gold set / Gold set`，仅按稳定 ID 匹配。标题匹配的候选单独报告为人工核验参考，不计入分子。零命中是实测 0（检索有问题），没有执行检索才是不可评估。 |
| 默认标准 | 叙事/范围：≥ 0.70；系统：≥ 0.85；快速：≥ 0.60；伞式：≥ 0.80 |
| 标准依据 | 检索策略验证惯例——系统综述的检索式应该能找回 85% 以上的已知锚点。叙事和范围综述允许较低的检索灵敏度，因为它们通常用引文跟踪和专家知识来补充。 |
| 何时可调整 | 你的检索策略故意很窄（如只查某会议论文）→调低。使用了多个互补数据库→某些数据库可对应调低。理由是"检索策略限制已说明"。 |
| 不达标意味着什么 | 检索词或数据库选择有盲区——可能错过大量相关工作。首先应检查是否是术语/缩写/同义词问题。 |
| 不可评估时怎么办 | 需要 Gold set（已知应命中的文献列表）和检索命中快照。AI 可通过 `search_for_eval.py` 自动生成诊断性检索，但这个 A2 是初步的——如果 gold 来自 A1 基准集，A1 和 A2 不独立，应标注为低证据强度。 |
| 证据类型 | 实测（可复跑检索+逐条命中）；估计（标题候选匹配）；不可评估（缺少 gold set 或检索快照） |

### A3 多源候选下界

| 项目 | 内容 |
|---|---|
| 易懂名称 | 多来源检索能发现的最少不重复候选数 |
| 它回答的问题 | 在既定范围内，来源之间去重后"至少有多少篇相关文献存在"？它不能直接告诉你"漏了多少"。 |
| 计算/核验 | 至少两个来源的检索快照按稳定 ID 去重并集。Capture–recapture 仅在来源近似独立的假设成立时可用（需要文档记录）。一般情况下只报告多源并集下界 + 灵敏度分析。 |
| 默认标准 | 至少两个完整来源；只报告下界，不设硬阈值 |
| 标准依据 | 多源并集是"存在"的近似下界，永远小于等于真实文献量。单源 count 不是覆盖率。 |
| 何时可调整 | N/A——A3 是描述性指标，不以"通过/不通过"判定。 |
| 不达标意味着什么 | N/A——A3 没有"达标"概念。只有一个来源→A3 不可评估。 |
| 不可评估时怎么办 | 需要至少两个来源的检索快照。AI 可通过 `collect_open_sources.py` 生成。若只有单一数据库（如用户限制），A3 标注 `not_assessable`，不能推断多源覆盖。 |
| 证据类型 | 估计（完整快照下界）；部分快照（不完整来源）；不可评估（单源） |

## B 饱和度——检索是否还在增长

### B1 核心库增长率（GGR）

| 项目 | 内容 |
|---|---|
| 易懂名称 | 每一轮新检索让库大了多少 |
| 它回答的问题 | 如果再搜一轮，会新增多少高置信纳入文献？连续两轮几乎不长了→检索可能接近饱和。 |
| 计算/核验 | `每轮高置信新增纳入数 / 该轮开始前核心库规模`。分子只计入经筛选确认的纳入文献——发现候选不等于纳入项，引用次数不等于筛选决定。 |
| 默认标准 | 最后两轮均 < 0.02（2%） |
| 标准依据 | 文献检索饱和文献中 GGR < 2% 是常用的收敛判据。但仅低 GGR 不够——需要 B2 和 B3 同时成立。 |
| 何时可调整 | 研究问题非常窄→自然增长少，2% 可能太严→调至 0.03 或 5%。系统综述→从严。 |
| 不达标意味着什么 | 检索可能尚未完成——可能存在未发现的文献簇。这不代表不能写综述，但应明确注明"检索未趋稳"。 |
| 不可评估时怎么办 | 固定稳健检索式至少两轮 `saturation_rounds`，每轮含 `core_before` 和经筛选确认的 `included_high`。q0→q* 检索式迭代不计入 B1/B2。无饱和度轮次时 B1/B2 不可评估；自动初筛不是饱和结论，需摘要/全文筛选与人工抽查升级。 |
| 证据类型 | 实测（有 ≥ 2 轮筛选确认的纳入记录）；candidate_discovery（仅候选发现，需筛选确认）；不可评估（无数据） |

### B2 新增路径发现率（DRR）

| 项目 | 内容 |
|---|---|
| 易懂名称 | 新路径带来"此前从未见过"的文献的比例 |
| 它回答的问题 | 新增一种检索方式（新数据库/前向引文/后向引文/新术语）后，是否发现了大量新文献？如果各路径都接近零→当前策略已覆盖所有已知检索路径。 |
| 计算/核验 | `路径 i 新发现的纳入文献 / 该路径筛选后候选数`。须保留每条路径的原始命中集，按预先声明顺序去重，不能因全局去重顺序让 DRR 失真。 |
| 默认标准 | 各路径均 < 0.05（5%） |
| 标准依据 | 与 GGR 互补——GGR 看"总量还在长吗"，DRR 看"换个角度搜还有惊喜吗"。 |
| 何时可调整 | 只有两条路径→3% 可能比 5% 更现实。有 5+ 路径→5% 能更好地区分真正的残差。 |
| 不达标意味着什么 | 某些检索路径仍有显著边际贡献——可能需要补充那个方向。 |
| 不可评估时怎么办 | 至少两条路径且每条有 `candidates`/`new_high_confidence`/`yield`。AI 自动检索阶段的路径产出的是发现候选而非纳入项→DRR 处于发现阶段，需筛选后方可评估。 |
| 证据类型 | 实测（有筛选确认）；candidate_discovery（仅候选发现）；不可评估（无数据） |

### B3 饱和过程证据

| 项目 | 内容 |
|---|---|
| 易懂名称 | 有没有独立验证来确认检索真的够了 |
| 它回答的问题 | 低 GGR 低 DRR 就够了？不——如果没有独立验证和路径完成证据，低增长可能只是检索策略共谋的盲区。 |
| 计算/核验 | 所有计划路径已完成 + 独立验证未发现漏项 + B1/B2 同时满足。 |
| 默认标准 | 路径完成率 = 1.0，独立验证通过 |
| 标准依据 | 检索饱和方法论文献——同一策略的反复检索会被同一盲区限制，独立验证是区分"真趋稳"和"策略共谋"的关键。 |
| 何时可调整 | 独立验证形式因领域而异——可以是外部专家评审、引用跟踪抽查或非检索来源比照。 |
| 不达标意味着什么 | 不能声称检索趋稳——必须在报告中明确"尚未证明稳定"，并给出补做建议。 |
| 不可评估时怎么办 | 独立验证通常需人工完成——AI 自动初审可能无法提供。此时 B3 标 `not_assessable`，B 维综合判定不能为"趋稳"。 |
| 证据类型 | 实测（有独立验证记录）；不可评估（缺验证或路径未完成） |

## C 平衡——主题与来源是否偏斜

### C1 主题覆盖与偏斜

| 项目 | 内容 |
|---|---|
| 易懂名称 | 各子方向/主题的文献分布是否均衡 |
| 它回答的问题 | 是否所有预期主题都有文献？某个主题是否严重挤占其他主题？空主题表示完全遗漏某个子方向。 |
| 计算/核验 | 统计各预期主题的文献数→计算最大主题占比、变异系数 (CV)、Gini、归一化 Shannon 熵。有 `target_share` 时再算 TVD。 |
| 默认标准 | 无空主题；Top ≤ 0.70；CV ≤ 0.80；Gini ≤ 0.50；Shannon ≥ 0.55；TVD ≤ 0.25（可选） |
| 标准依据 | 基于 Shannon 信息论和 Gini 不平等度量在文献计量中的经验共识——这些阈值不是绝对标尺，而是"检查一下这个方向"的信号。 |
| 何时可调整 | 你的视角故意聚焦某子方向→可以声明哪几个主题是核心，其余为背景；此时 Top 可以 > 0.70 且不报警。 |
| 不达标意味着什么 | 空主题→完全遗漏；高偏斜→综述可能视野失衡，应在局限中声明。对立观点主题缺文献→可能只收录了支持假设的文献（cherry-picking 风险）。 |
| 不可评估时怎么办 | 需要 taxonomy（主题列表 + 每主题文献数）。可以标注 `opposing_viewpoint: true` 来标记含对立/竞争观点的主题——这不是必需的，但 C1 会据此附加定性提示。如果无法自动分类，AI 可能要求人工提供分类。没有 taxonomy→C1 不可评估。 |
| 证据类型 | 实测（有外部分类采集数据）；自动初筛（AI 从标题推断主题）；不可评估（无 taxonomy） |

### C2 来源集中度

| 项目 | 内容 |
|---|---|
| 易懂名称 | 文献是否过度依赖单一数据库/来源 |
| 它回答的问题 | 如果 90% 的文献都来自 OpenAlex 而完全没查 IEEE Xplore，工程综述可能遗漏大量实际研究。 |
| 计算/核验 | 统计各来源记录数→计算最大来源占比、CV、Gini、归一化 Shannon。 |
| 默认标准 | Top ≤ 0.80；CV ≤ 1.00；Gini ≤ 0.60；Shannon ≥ 0.45；来源 ≥ 3 时 Shannon ≤ 0.95（过于碎片化也需解释） |
| 标准依据 | 来源集中度是对检索策略覆盖广度的间接诊断——不是质量度量，而是偏倚风险信号。 |
| 何时可调整 | 如果你故意只用一个数据库并已声明限制，单一来源的高占比是策略后果而非缺陷。 |
| 不达标意味着什么 | 风险：可能存在数据库索引偏差——某个数据库不收录的期刊或会议可能被完全忽略。附加诊断：作者集中度（单一作者/课题组 >25%）提示可能过度依赖单一研究群体视角。 |
| 不可评估时怎么办 | 库中须有来源字段。Zotero 库通常自带来源信息；纯 PDF 目录可能需要手动标注。 |
| 证据类型 | 实测（有来源标注）；自动初筛（来源名未标准化）；不可评估（无来源信息） |

### C3 主题-来源交叉

| 项目 | 内容 |
|---|---|
| 易懂名称 | 每个子方向是否有独立、多样的来源支撑 |
| 它回答的问题 | 如果某个主题的文献全部来自同一个数据库，这个主题的覆盖可能被该数据库的索引偏差左右。 |
| 计算/核验 | 逐主题检查至少两个来源且单一来源占比 ≤ 0.80。 |
| 默认标准 | 每主题 ≥ 2 来源；单来源占比 ≤ 0.80 |
| 标准依据 | 主题×来源交叉是覆盖稳健性的最细粒度检查——某主题多来源→"覆盖健全"；某主题单来源→"可能漏掉不收那个数据库的研究"。 |
| 何时可调整 | 极窄主题可能确实只有一个主要来源——此时 C3 可以设 warning 而非 fail。 |
| 不达标意味着什么 | 该主题的覆盖可能受单个数据库收录政策的影响——检索策略可能需要针对该主题补充其他来源。 |
| 不可评估时怎么办 | 需要 `topic_source_counts` 输入。无数据→C3 不可评估。 |
| 证据类型 | 实测（有逐主题来源计数）；自动初筛；不可评估 |

### C4 观点偏斜度

| 项目 | 内容 |
|---|---|
| 易懂名称 | 围绕核心论点，是否只收集了同一方向的证据 |
| 它回答的问题 | 例如综述主张“方法 X 在场景 Y 中有效”时，库里是否也保留了无效、受限、失败或条件性结论，避免 cherry-picking。 |
| 计算/核验 | 先写明中心主张和去重后的被分类记录数，再把文献标为支持、质疑、条件性/混合或未分类；计算分类覆盖、支持/质疑比例和反方数量。 |
| 默认标准 | 分类覆盖 ≥50%；对可争议主张，单方方向性证据 ≤90%，反方至少 3 篇。 |
| 不达标意味着什么 | 不是证明某一方一定正确，而是说明当前库不足以检验反例与适用边界；应补检反向术语、失败条件和比较研究。 |
| 不可评估时怎么办 | 没有中心主张或立场分类时，C4 直接给 warning“无法检查观点偏斜”，不会从泛化的正负面词臆测学术立场。 |
| 证据类型 | AI 标题/摘要分类为 automated-screening；记录分类方法并抽样核验后可升级为实测。 |

### C5 综述间覆盖分布（仅伞式综述）

| 项目 | 内容 |
|---|---|
| 易懂名称 | 纳入的各篇综述在子主题、方法类型、覆盖年份上是否互补 |
| 它回答的问题 | 如果 5 篇纳入综述覆盖的都是同一组原始研究和同一子方向，伞式综述的增量价值有限。 |
| 计算/核验 | 纳入综述的子主题分布、方法类型分布（系统 vs 叙事 vs 范围）、检索窗口分布。CCA 衡量综述间原始研究重叠度。 |
| 默认标准 | CCA ≤ 0.15；子主题/方法类型无显著断层 |
| 标准依据 | Pieper 等关于伞式综述重叠的文献——CCA > 0.15 表示综述间重叠较高。 |
| 何时可调整 | CCA 计算需要纳入综述的原始研究列表——不可得时 C5 只能报告可得的分布信息。 |
| 不达标意味着什么 | 重叠过高→伞式综述的"整合"价值可能被高估；方法类型单一→缺乏多样证据强度锚点。 |
| 不可评估时怎么办 | CCA 可能不可算→仅报告方法类型和检索窗口的自动诊断。 |
| 证据类型 | 自动初筛（方法类型/检索窗口）；估计或不可评估（CCA） |

## D 时效——库是否反映当前状态

### D1 来源新鲜度

| 项目 | 内容 |
|---|---|
| 易懂名称 | 检索来源的最后检索日期距今多久 |
| 它回答的问题 | 如果最后一次搜 OpenAlex 是两年前，库必然有严重的时效缺口。 |
| 计算/核验 | 各来源最后一次成功检索日期距今天的天数。 |
| 默认标准 | AI/通信：30 天；常规工程：90 天；基础设施：180 天（由 profile 决定） |
| 标准依据 | 快变领域（AI/通信）的技术状态可能在数月内显著变化；土木/能源/航天有长期规范与可靠性数据的长尾。 |
| 何时可调整 | 如果你想用半年前的检索→调大天数，但在报告中承认时效衰退。 |
| 不达标意味着什么 | 库可能缺少近几个月的文献——如果是快变领域，这会构成实质缺口。 |
| 不可评估时怎么办 | 需要各来源的检索日期。没有→D1 不可评估。 |
| 证据类型 | 实测（有日期）；不可评估（无日期） |

### D2 近年文献比例

| 项目 | 内容 |
|---|---|
| 易懂名称 | 近年发表的文献占比是否足够 |
| 它回答的问题 | 综述是写给当下的读者——如果库中 80% 的文献超过 5 年，可能在一个快变领域中过时。 |
| 计算/核验 | 近 N 年记录数 / 有日期记录总数。年份字段完整率 < 50% 时自动降级为 `warning`（防止缺失数据掩盖真实比例）。 |
| 默认标准 | AI/通信：近 3 年 ≥ 40%；常规工程：近 5 年 ≥ 35%；基础设施：近 7 年 ≥ 30% |
| 标准依据 | 不同工程领域技术迭代速度的经验判断——这不是质量裁决，而是时效诊断。 |
| 何时可调整 | 历史综述→可以近为零；前沿快报→可从严至 60%。 |
| 不达标意味着什么 | 库可能偏向老文献——如果前沿是综述的一部分，需补充最新检索。 |
| 不可评估时怎么办 | 年份字段缺失过多→D2 降为 warning。无任何日期→不可评估。 |
| 证据类型 | 实测（有日期）；warning（年份完整率 < 50%） |

### D3 前沿覆盖

| 项目 | 内容 |
|---|---|
| 易懂名称 | 对新兴术语、新平台或新标准是否专门检索过 |
| 它回答的问题 | "大模型时代"的综述不能只用"deep learning"当关键词——有没有搜过"LLM""GPT""foundation model"？ |
| 计算/核验 | 新术语/新平台/新标准有独立检索记录或 gold set 命中证据。 |
| 默认标准 | 前沿窗口有独立检索证据或 gold set 覆盖 |
| 标准依据 | 近期发表 ≠ 前沿覆盖——库中近期文献多可能只是因为在一个常规检索式里碰巧命中。前沿需要显式的术语/来源覆盖验证。 |
| 何时可调整 | 如果不是前沿综述→D3 可以放宽要求。 |
| 不达标意味着什么 | 可能错过最近该领域的命名突破——这不是"库过时"，而是"覆盖没跟上命名"。 |
| 不可评估时怎么办 | 提供前沿检索记录或 gold set。默认 `not_assessable` 不是失败。 |
| 证据类型 | 实测（有前沿检索快照）；不可评估 |

### D4 版本区分

| 项目 | 内容 |
|---|---|
| 易懂名称 | 预印本和正式出版版是否已区分、不重复计数 |
| 它回答的问题 | 同一篇文章的 arXiv 版和期刊正式版如果在库里出现两次→结论可能被双计数影响。 |
| 计算/核验 | 预印本—正式版、会议版—期刊版关系已核验；重复不计数。 |
| 默认标准 | 版本关系已核验 |
| 标准依据 | 工程领域 arXiv 预印本普遍——版本合并是去重的基本步骤。 |
| 何时可调整 | N/A——版本区分是所有综述类型的基础需求。 |
| 不达标意味着什么 | 存在重复计数风险——C/D/E 维中任何涉及计数的指标都可能被歪曲。 |
| 不可评估时怎么办 | 需要版本核验记录或 dedup-log。默认 `not_assessable`。 |
| 证据类型 | 实测（有 dedup-log）；自动初筛（仅 DOI 去重）；不可评估 |

## E 学术影响——仅背景参考

E1（h-core）和 E2（Tier-1 覆盖）都不设硬阈值。它们只回答两个辅助问题："核心文献被引得多吗"和"顶级场所覆盖了多少"。h-core 对库规模、学科引用习惯和新发表论文极敏感——高 h-core ≠ 高质量，低 h-core ≠ 低质量。真正的研究质量评估应使用 AMSTAR-2、ROBIS 或与研究设计匹配的批判性评价工具。

## F 可用性——文献能否拿来写综述

### F1 检索可复跑

| 项目 | 内容 |
|---|---|
| 易懂名称 | 检索过程是否能被第三人在另一台电脑上复跑 |
| 它回答的问题 | "我用某个数据库搜过"——能重现吗？ |
| 计算/核验 | run-log 文件包含查询原文、来源、字段 (fields)、日期、过滤器 (filters)、结果数。缺返回信息算不完整。 |
| 默认标准 | 查询原文、字段、日期、来源、过滤器齐全（`source, query, fields, date` 为最小必需；`filters, result_count, completion_status` 为推荐字段） |
| 标准依据 | 可复跑是可复现综述的最低标准——缺少查询原文和日期是唯一过程阻断项。 |
| 何时可调整 | N/A——所有综述类型都需要检索可复跑。 |
| 不达标意味着什么 | 查询原文缺失→检索不能被复跑→综述的可信度受限。这是报告中唯一的过程阻断项。 |
| 不可评估时怎么办 | 提供 run-log 文件。没有→不可评估，不是失败——但应在报告中注明。 |
| 证据类型 | 实测（有完整 run-log）；不可评估（缺文件或文件不完整） |

### F2 摘要覆盖率

| 项目 | 内容 |
|---|---|
| 易懂名称 | 多少条目有摘要可以用于自动初筛 |
| 它回答的问题 | C/E 维的自动评估依赖摘要信息——如果只有 30% 有条目有摘要，自动初筛的可靠性会下降。 |
| 计算/核验 | 有摘要条目数 / 库总条目数 |
| 默认标准 | 叙事/范围：≥ 0.80；系统：≥ 0.85；快速：≥ 0.70；伞式：≥ 0.80 |
| 标准依据 | 元数据完整性对自动筛选和指标计算的最低要求。 |
| 何时可调整 | 如果全部人工筛选→摘要率要求可降低。 |
| 不达标意味着什么 | 自动初筛的可靠性下降——C/E 维的自动判定置信度应随之下调。 |
| 不可评估时怎么办 | 库中需有摘要字段。数据缺失→自动提取或人工补充。 |
| 证据类型 | 实测（有摘要率）；不可评估（无数据） |

### F3 全文获取率

| 项目 | 内容 |
|---|---|
| 易懂名称 | 多少文献有 PDF 附件或开放链接 |
| 它回答的问题 | 写综述需要读全文——如果大部分文献只有标题没有全文，综述的深入程度受限制。 |
| 计算/核验 | 有附件或开放链接任一可用的记录数 / 总记录数（联合比例，非相加——避免同一记录双渠道重复计数） |
| 默认标准 | 叙事/范围：≥ 0.60；系统：≥ 0.80；快速：≥ 0.50；伞式：≥ 0.85 |
| 标准依据 | 系统综述需要逐篇全文核对；叙事综述按需取阅论文；伞式综述的全文本要求最高——AMSTAR-2/ROBIS 评估必须逐篇阅读全文。 |
| 何时可调整 | 图书馆有订阅访问但未反映在条目中→实际情况可能比 F3 显示的好。 |
| 不达标意味着什么 | 写综述时可能面临全文获取瓶颈——这不影响库的覆盖充分性，但影响你实际能用到多少文献。 |
| 不可评估时怎么办 | 库中需有附件或开放链接字段。Zotero 库通常自带。 |
| 证据类型 | 实测（有附件/链接率）；估计（仅开放链接）；不可评估（无数据） |

### F4 去重与版本

| 项目 | 内容 |
|---|---|
| 易懂名称 | DOI 完全相同的是否已合并；预印本-正式版等版本冲突是否已处理 |
| 它回答的问题 | 同一篇文献在库中是否有多个记录？ |
| 计算/核验 | DOI 精确重复 = 0；版本候选（标题年份相似但 ID 不同）已有决策（合并/保留两者/排除/待人工核验）。 |
| 默认标准 | 无 DOI 精确重复；版本候选有决定和理由 |
| 标准依据 | 去重是所有文献计量的前提——重复计数会扭曲所有基于计数的指标。 |
| 何时可调整 | N/A——所有综述类型都需要去重。 |
| 不达标意味着什么 | 存在未处理重复→C/D/E 维的计数可能不准确。 |
| 不可评估时怎么办 | 需要 dedup-log。没有→标 `not_assessable`。 |
| 证据类型 | 实测（有结构化 dedup-log 且决策完整）；not_assessable（无 log 或无决策） |

**附加诊断——DOI/书目一致性**：F4 只检查库内重复，不检查每条记录的 DOI 是否能在 Crossref/PubMed 查到、author/year/title 是否与来源一致。**如果使用了 Zotero/EndNote 等自动抓取的元数据，建议人工抽查 10–20 条记录的 DOI 有效性和著录完整性**，以免 Garbage In, Garbage Out 扭曲 C/D/E 维的自动判定。完整的引文准确性验证属于未来可做的 `review-manuscript-audit` skill 范围，本 skill 仅提供此项提示。默认为 `not_assessable`（无专门核验记录）。

### F5 来源可追溯

| 项目 | 内容 |
|---|---|
| 易懂名称 | 多少条记录可以追溯到哪个检索批次或来源 |
| 它回答的问题 | 这个库里每篇文章是哪次检索来的？有没有纳入/排除理由？ |
| 计算/核验 | 有来源谱系的核心记录比例 |
| 默认标准 | 叙事/范围：≥ 0.85；系统：≥ 0.95；快速：≥ 0.70；伞式：≥ 0.90 |
| 标准依据 | 可追溯是可复现综述的核心要素——系统综述要求最高。 |
| 何时可调整 | 历史积累的库→可以部分记录无法追溯到原始检索。 |
| 不达标意味着什么 | 库的构建过程部分不可复现——这不影响内容质量，但降低可复现性。 |
| 不可评估时怎么办 | 库中需有来源或检索批次字段。 |
| 证据类型 | 实测（有谱系率）；不可评估（无来源字段） |

### F6 撤稿更正核查

| 项目 | 内容 |
|---|---|
| 易懂名称 | 关键记录是否已核查撤稿/更正状态 |
| 它回答的问题 | 库里有没有已经被撤稿的论文？ |
| 计算/核验 | 关键记录的撤稿/更正状态已通过专门来源核验。 |
| 默认标准 | 需专门来源核验——自动检测不完备 |
| 标准依据 | 撤稿信息通常不在文献数据库中完整更新，需要主动核查。 |
| 何时可调整 | N/A——所有综述类型都应避免引用已被撤稿的论文。 |
| 不达标意味着什么 | 存在引用已被撤稿文献的风险。 |
| 不可评估时怎么办 | 需要专门来源核验或 Retraction Watch 等工具。默认 `not_assessable` 是常态。 |
| 证据类型 | 人工核验；不可评估（未经专门核查） |

### F7 综述质量评估就绪度（仅伞式综述）

| 项目 | 内容 |
|---|---|
| 易懂名称 | 是否有足够全文来做 AMSTAR-2/ROBIS 质量评估 |
| 它回答的问题 | 伞式综述必须对纳入的每篇综述做方法学质量评估——没有全文就没法做。 |
| 计算/核验 | 纳入综述中具备全文可进行质量评估的比例 + 质量评估工具选定状态 |
| 默认标准 | 全文就绪度足以支撑评估（≥ 0.85）；已选定 AMSTAR-2 或 ROBIS |
| 标准依据 | AMSTAR-2 的 16 项评分需要逐项阅读全文——全文缺口 = 评估缺口。 |
| 何时可调整 | N/A——伞式综述的质量评估是方法学必需项。 |
| 不达标意味着什么 | 部分纳入综述无法完成质量评估→伞式综述结论的信度受限。 |
| 不可评估时怎么办 | 逐篇标注全文获取状态和质量评估工具选择。F7 只报告就绪度，不代替 AMSTAR-2 评分。 |
| 证据类型 | 自动初筛（全文状态）；需人工完成（AMSTAR-2 评分） |

## 如何使用这份说明书

1. 首次评估时，**AI 会默认采用上述标准**，并结合你的综述类型和工程领域选择对应的阈值。
2. **你可以覆盖任意阈值**。只需说"A1 我只要 0.65 就行，因为我的题目非常窄"。
3. **报告会包含"本次采用标准"附录**，列出实际生效的每一个阈值、其来源（默认/用户覆盖），以及你确认的日期。
4. **`not_assessable` 不是失败**。它是"尚缺输入"——用来指导下一步，不是批评。
5. **每项结论都标有证据类型**：实测的你可以信任；估计的需要谨慎解读；自动初筛的必须人工核验；不可评估的需要补输入。
