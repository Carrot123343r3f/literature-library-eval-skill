# 工程文献库审计 Skill 使用说明

## 如何阅读统一审计表

报告以一张 A–F 平级总表呈现所有 A1–F8 子项。每一行同时写明标准、是否达标、当前测量/观察、证据状态与下一步行动。`not_assessable` 表示目前没有足以作出该项判断的输入或可复跑证据，并不等于失败；它要求补充相应日志、快照、人工核验或项目阈值。

各子项的通俗报告名称、含义、输入字段和计算/核验方法见 [指标字典](references/indicator-dictionary.md)。报告中使用该字典的名称；内部代码仍保留 A1–F8 作为稳定编号。
`literature-library-eval` 用于判断一套工程研究文献库能否支撑指定综述，不用于给论文“打质量分”。支持计算机与 AI、电子通信、机械制造、土木建筑、材料、能源、环境、化工过程、航空航天、交通与生物医学工程；不处理纯数学、纯物理、纯化学、临床医学或基础生命科学。

## 它会输出什么

- A1：已确认必纳入工程文献是否在库中。
- A2：检索式是否能找回已知相关工作；失败检索会明确显示为 0。
- A3：多源候选集的覆盖下界或带假设的区间，不会用库条数冒充召回率。
- 工程关键层、工况、方法、基线、指标和应用场景的缺口。
- 检索趋稳证据、来源级检索新鲜度、版本/重复/访问/追溯风险。
- 可复跑的 Markdown、HTML、JSON 报告及查询、来源和决策日志。

报告不会只给 A–F 的总标签。A–F 是平级维度：A 已知文献找回情况，B 检索是否趋于饱和，C 库的来源与主题是否平衡，D 库是否反映当前研究状态，E 库是否包含足够的学术质量信号，F 文献是否真正可用于写综述。默认阈值及对快速领域、基础设施或系统综述的调整都会写入报告。

## 首次调用

例如：

> 使用 literature-library-eval 审计我的 Zotero 文献库，判断它能否支持关于工业视觉缺陷检测的系统综述。

AI 会一次确认研究问题、综述类型、纳入范围、时间与语言限制、文献库位置、自动化方式、可用数据源和输出格式。之后会自主检索、调词、去重、筛选与报告；只有必要输入缺失、来源不可访问或范围改变时才会暂停。

## 如何解读

每项结果都会标为：**实测**、**估计**、**自动初筛**、**待人工核验**或**不可评估**。“在声明范围内趋稳”不等于收齐全球所有文献；引用量、期刊等级、近期论文比例和附件数量也不等于质量。

默认使用开放来源；如设备已有合法订阅访问，才会增强使用工程领域数据库。请通过已配置的连接或环境变量提供凭据，不要在对话中粘贴 API key。

## 迁移

复制整个 skill 文件夹即可。Codex 的用户级路径通常是 `~/.codex/skills/literature-library-eval`，Claude 通常是 `~/.claude/skills/literature-library-eval`。从 GitHub 获取后保留目录结构，并重新打开应用或创建新任务。
