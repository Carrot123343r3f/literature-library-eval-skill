# 文献库评估 Skill 使用说明

## 如何阅读评估总表

报告以一张 A–F 六维评估总表呈现所有 A1–A3、B1–B3、C1–C3、D1–D4、E1–E2、F1–F6（共 21 行）。每一行同时写明标准、判定、当前值、证据状态与说明行动。`not_assessable` 表示目前没有足以作出该项判断的输入或可复跑证据，并不等于失败；它要求补充相应日志、快照、人工核验或项目阈值。

各子项的通俗报告名称、含义、输入字段和计算/核验方法见 [指标字典](references/indicator-dictionary.md)。报告中使用该字典的名称；内部代码保留 A1–F6 作为稳定编号。
`literature-library-eval` 用于判断一套工程研究文献库能否支撑指定综述，不用于给论文"打质量分"。支持计算机与 AI、电子通信、机械制造、土木建筑、材料、能源、环境、化工过程、航空航天、交通与生物医学工程；不处理纯数学、纯物理、纯化学、临床医学或基础生命科学。

## 它会输出什么

报告含**七个段落**（`audit.md` / `audit.html` / `audit.json`）：

1. **基本信息**——生成时间、评估对象、库规模、综述类型、工程领域、研究范围、全域候选下界
2. **综合判断**——叙述性总结：达标/阻断/关键数字，明确"不可评估不是失败"
3. **评估方法与过程**——把 agent 的隐式工作显式化：检索关键词与检索式、A1 基准集与 A2 Gold set 的构建方法、数据来源、分类体系、证据状态图例
4. **A–F 六维评估总表**——21 行（A1–F6），每行含标准、判定、当前值、证据状态、说明与行动
5. **各维度分析**——A/B/C/D/E/F 六个维度各一段叙述，解释数字含义和根因
6. **改进建议**——阻断项与建议改进分列，每项附原因和定向
7. **局限与声明**——诚实声明：什么能测、什么不能、哪些是估计/自动初筛

A–F 六个维度不合成总分：A 覆盖（已知文献找回）、B 饱和度（检索是否趋稳）、C 平衡（主题与来源是否偏斜）、D 时效（库是否反映当前状态）、E 学术影响（h-core/Tier-1，仅背景信号）、F 可用性（摘要/PDF/去重/谱系/撤稿）。默认阈值及对快速领域、基础设施或系统综述的调整都会写入报告。

## 首次调用

例如：

> 使用 literature-library-eval 评估我的 Zotero 文献库，判断它能否支持关于工业视觉缺陷检测的系统综述。

AI 按 `references/confirmation-template.md` 的**四批次**完成一次确认：批次一（研究问题/综述类型/库位置，必须逐项拍板）→ 批次二（自主策略/数据源/范围推断/输出格式，有推荐默认可一键跳过）→ 批次三（补充材料：已知必纳入文献、已有检索词、分类体系、已知缺口、权威综述，可选但会提示）→ 批次四（合法性声明，告知不征求）。确认后自主检索、调词、去重、筛选与报告；只有必要输入缺失、来源不可访问或范围改变时才暂停。

## 如何解读

每项结果都会标为：**实测**、**估计**、**自动初筛**、**待人工核验**或**不可评估**。“在声明范围内趋稳”不等于收齐全球所有文献；引用量、期刊等级、近期论文比例和附件数量也不等于质量。

默认使用开放来源；如设备已有合法订阅访问，才会增强使用工程领域数据库。请通过已配置的连接或环境变量提供凭据，不要在对话中粘贴 API key。

## 迁移

复制整个 skill 文件夹即可。Codex 的用户级路径通常是 `~/.codex/skills/literature-library-eval`，Claude 通常是 `~/.claude/skills/literature-library-eval`。从 GitHub 获取后保留目录结构，并重新打开应用或创建新任务。
